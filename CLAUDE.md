# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

A purpose-built, single-page expert evaluation survey for a multilingual patent intelligence automation system. N=13 pilot study with 2 independent blind evaluators (Traditional Chinese interface). There is no build process — the application is two files served directly in a browser.

## Running the Application

Open `index.html` directly in a browser, or serve it with any static file server:

```bash
python3 -m http.server 8000
# then open http://localhost:8000
```

There are no install steps, no build steps, no tests, and no linting configuration.

## Architecture

### File Roles

- **`index.html`** — The entire application: all CSS (inline `<style>`), all JavaScript (inline `<script>`), and HTML structure. ~1200 lines.
- **`items.js`** — Auto-generated data file. Sets `window.ITEMS` (array of 13 patent evaluation items). **Do not edit by hand** — it is regenerated by `export_items.py` in the source data repository.

### State Management

All application state lives in a single `state` object (defined around line 258 of `index.html`). State is written to `localStorage` on every field change via `save()`. The localStorage key is `expert_eval_{expertId}`, so Expert A and Expert B have independent, non-overlapping state.

On load, state is restored from localStorage if present; otherwise initialized fresh.

### Screen Flow

Four screens controlled by `state.screen` index into `SCREENS = ["intro","expert","item","done"]`. Navigation is via `go(dir)` which validates required fields before advancing. The `REQUIRED` constant (around line 254) lists mandatory field names per screen.

### Item Ordering

Items are presented in a seeded-random order per expert, using a mulberry32 PRNG. Seeds are defined in the `SEED` constant (`SEED.A = 101`, `SEED.B = 202`). This ensures reproducible, divergent orderings between the two experts without server coordination.

### Evaluation Dimensions

Eight questions per item (Q1–Q8) covering:
- **Q1** Translation adequacy (5-point Likert + N/A)
- **Q2** Summarization faithfulness (5-point Likert + N/A)
- **Q3** Hallucination (Yes/No/N/A)
- **Q4** Entity extraction TP/FP/FN counts with N/A toggle
- **Q5** Document classification (6 categories + N/A) — **blind**: AI verdict hidden
- **Q6** Actionability (3 levels + N/A) — **blind**: AI verdict hidden
- **Q7** Format compliance (Pass/Fail)
- **Q8** Overall usefulness (5-point Likert + N/A)

Q5 and Q6 intentionally hide the system's verdicts from the expert during evaluation. This is enforced in `renderItem()` — do not accidentally expose those fields.

### Data Submission

`submitData()` POSTs a JSON payload to `CONFIG.submitEndpoint` (a Google Apps Script web app URL). The `flatRows()` function serializes the nested state into tabular rows. Experts can also download raw JSON or CSV locally at any time.

### Timing

Per-item response time is tracked automatically. A timer starts when an item is first displayed and is accumulated in `state.timings[itemId]` (seconds).

## Key Conventions

- **Global scope**: All functions and constants are global (`window.*`). There is no module system.
- **No framework**: Plain ES5-compatible JavaScript; no transpilation assumed.
- **`items.js` is read-only**: If evaluation items need to change, regenerate via the upstream Python script, then replace the file wholesale.
- **Expert identity matters**: The expert selection screen (screen index 1) determines the localStorage key and item ordering seed. Changing an expert's selection mid-survey would corrupt their state.
- **`CONFIG.submitEndpoint`**: The only external dependency. If the Apps Script URL changes, update this constant near the top of the `<script>` block in `index.html`.
